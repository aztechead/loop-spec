# Pause

Invoked as `/super-spec:pause`. Captures the current feature state into two crash-recovery artifacts so work can resume safely in a later session.

## Inputs

- `feature_path` (optional): path to `.super-spec/features/{slug}/feature.json`. Defaults to scanning `.super-spec/features/*/feature.json` for the active feature.

## Procedure

### Active workflow snapshot

Before snapshotting feature.json, if `feature.json.activeWorkflow` is set:

```bash
RUN_ID=$(jq -r '.activeWorkflow.runId // ""' .super-spec/features/{slug}/feature.json)
# Stop the workflow if /workflows shows it as running (advisory; runtime
# already handles session exit gracefully).
```

Persist `{scriptPath, args, sessionId, runId, startedAt}` into HANDOFF.json
under `activeWorkflow`. Note: `runId` is informational only; cross-session
resume will re-dispatch with `scriptPath + args` and receive a new runId.

### Step 1 - Run pause-snapshot

```bash
bash "${CLAUDE_SKILL_DIR}/../../lib/pause-snapshot.sh" [--feature-dir <path>]
```

`lib/pause-snapshot.sh` generates both artifacts atomically into `.super-spec/features/{slug}/`. Kill switch: if `SUPER_SPEC_PAUSE=0` is set the script exits 0 without writing any file.

### Step 2 - Print summary

After the script exits 0, print to the user:

- Path to `HANDOFF.json`
- Path to `.continue-here.md`
- Current phase and pending task count (read from HANDOFF.json)
- Any blocking constraints listed in `.continue-here.md`

Example output:

```
Pause snapshot written.
  HANDOFF.json:      .super-spec/features/{slug}/HANDOFF.json
  .continue-here.md: .super-spec/features/{slug}/.continue-here.md
  Current phase:     execute
  Pending tasks:     2
  Blocking:          3 constraints listed in .continue-here.md
```

### Step 3 - Exit worktree (schema-6 features only)

If the feature's `feature.json` contains a `worktreePath` field (schemaVersion 6), call:

```
ExitWorktree({ action: "keep" })
```

This returns the session to the main checkout while the feature worktree and branch `feat/{slug}` remain on disk for later resume. Skip this step for legacy features (schemaVersion <= 5) that have no `worktreePath`.

## HANDOFF.json schema

`HANDOFF.json` contains seven keys. All keys are always present; missing data is represented as an empty string or empty array.

| Key | Type | Description |
|-----|------|-------------|
| `currentPhase` | string | Value of `feature.json.currentPhase` at snapshot time (e.g., `"execute"`, `"verify"`). |
| `completedTasks` | array | Tasks or phases marked complete before the pause (sourced from `feature.json.completedPhases` or `completedTasks`). |
| `pendingTasks` | array | Tasks still outstanding (sourced from `feature.json.pendingRemediationTasks`). |
| `blockers` | array | Any blockers recorded in `feature.json.blockers`. |
| `decisions` | array | Architectural decisions recorded in `feature.json.decisions`. |
| `uncommittedFiles` | array | Files reported by `git diff --name-only HEAD` at snapshot time. Empty array when the working tree is clean. |
| `contextNotes` | string | Human-readable advisory note including the snapshot timestamp and a pointer to `.continue-here.md`. |

Example:

```json
{
  "currentPhase": "execute",
  "completedTasks": ["discuss", "plan"],
  "pendingTasks": ["fix: test suite regression"],
  "blockers": [],
  "decisions": [],
  "uncommittedFiles": ["lib/foo.sh"],
  "contextNotes": "Snapshot generated by lib/pause-snapshot.sh at 2026-05-28T12:00:00Z. Review BLOCKING CONSTRAINTS in .continue-here.md before resuming."
}
```

## .continue-here.md format

`.continue-here.md` is a human-readable resume guide structured into three sections.

### BLOCKING CONSTRAINTS

A checklist of items that must be resolved before auto-resuming. Each item is tagged `blocking:`.

```
- [blocking: verify HANDOFF.json is current before dispatching any task]
- [blocking: confirm feature branch is up to date with base before writing new commits]
- [blocking: if uncommittedFiles is non-empty, stage or stash before resuming]
```

Severity: **blocking**. The cycle/SKILL.md resume detection reads these tags and prevents automatic re-dispatch until all blocking constraints are cleared.

### ANTI-PATTERNS

Patterns that have caused problems in prior sessions. Items carry one of two severity tags:

- `blocking:` - must be avoided before resuming; auto-resume is held.
- `advisory:` - informational warning; auto-resume may proceed with caution.

```
- [advisory: do not skip the verify command when marking a task complete]
- [advisory: do not batch multiple task commits without running tests between them]
- [blocking: do not push or merge without the verifier agent sign-off]
```

### REQUIRED READING

An ordered list of files to read before writing any code in a resumed session.

```
1. HANDOFF.json (this feature dir) - current phase, pending tasks, blockers
2. docs/super-spec/features/{slug}/PLAN.md - full task DAG and acceptance criteria
3. feature.json (this feature dir) - gate history, retry budget, branch info
```

## Severity-tag parsing for cycle resume

`skills/cycle/SKILL.md` reads `.continue-here.md` when resuming a paused feature. It scans lines for `[blocking:` and `[advisory:` tags:

- If any `[blocking:` line is found: auto-resume is held. The lead presents the blocking constraints to the user and requests explicit acknowledgment before dispatching any task.
- If only `[advisory:` lines are found: auto-resume proceeds. The advisory items are printed as warnings before the first task is dispatched.
- If `.continue-here.md` is absent: resume proceeds normally (treated as no constraints).

This parsing allows the severity level written at pause time to directly control the re-entry posture at resume time, without requiring a separate configuration file.
