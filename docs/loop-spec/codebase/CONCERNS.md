## Security Findings

### Hook transcript path injection (medium)

**File:** `/Users/cbobrowitz/Projects/loop-spec/hooks/restrict-agent-paths.sh:29`

`TRANSCRIPT_PATH` is extracted from untrusted hook JSON input and passed as `sys.argv[1]` directly to an inline Python script. If a crafted `transcript_path` value pointed to a world-writable or attacker-controlled file, the hook would read from it and derive an incorrect `CALLER` identity, potentially bypassing the path-restriction gate. The hook runs with the same privileges as the Claude Code process.

Mitigation present: the Python block catches `Exception` broadly and defaults `caller = ""` (unrestricted) on any read failure. This is a safe fallback but silently neutralizes the hook on read error.

### Hook allows all unknown subagent_types unrestricted (low)

**File:** `/Users/cbobrowitz/Projects/loop-spec/hooks/restrict-agent-paths.sh:104-107`

The final `*)` case allows any subagent type not explicitly named. A new agent definition added to the repo without updating this hook will get unrestricted write access. The comment says "defensive default" but this is a permissive default, not a restrictive one. As agent catalog grows, this is a latent escape hatch.

### validate-agents runs with relative paths (low)

**File:** `/Users/cbobrowitz/Projects/loop-spec/tests/validate-agents.sh:8`

`ls agents/loop-spec-*.md` uses a relative path with no `cd` guard. The test must be run from repo root; run from any other directory silently passes (no matching files, `EXPECTED=14` check catches it, but the error message is confusing). Not a security risk but a fragile test harness assumption.

### state.json stored in an untracked, not gitignored-at-root location (low)

**File:** `/Users/cbobrowitz/Projects/loop-spec/.gitignore:1-2`

`.loop-spec/features/` and `.loop-spec/worktrees/` are gitignored, which is correct. However, `.loop-spec/codebase/index.json` is explicitly NOT gitignored (noted in `skills/map-codebase/SKILL.md:84`). That file tracks per-file-to-domain mappings. If a consuming project has sensitive filenames (e.g., `secrets/prod.yaml`), those paths will be committed into `index.json` in `docs/loop-spec/codebase/`. Low risk for the plugin repo itself, relevant for adopters.

## Risky Patterns

### Python3 and jq are undeclared runtime dependencies (medium)

**Files:**
- `/Users/cbobrowitz/Projects/loop-spec/hooks/restrict-agent-paths.sh:21,28,32`
- `/Users/cbobrowitz/Projects/loop-spec/lib/state-write.sh:32`
- `/Users/cbobrowitz/Projects/loop-spec/tests/smoke.sh:3`

The project declares "Zero deps. No npm, pip, brew." in the README but `python3` and `jq` are hard runtime requirements for the hook and state-write library. `python3` is used for JSON parsing in the hook; `jq` is used for JSON validation in `state-write.sh` and for all state assertions in `smoke.sh`. Neither is verified at startup. On a system where one is absent, the hook silently falls back to `CALLER=""` (unrestricted) and `state-write.sh` rejects all writes with exit 1 (fails loudly). These should be documented or checked at startup.

### Inline git operations in skill markdown prose (medium)

**Files:** `/Users/cbobrowitz/Projects/loop-spec/skills/execute/SKILL.md`, `/Users/cbobrowitz/Projects/loop-spec/skills/verify/SKILL.md`, `/Users/cbobrowitz/Projects/loop-spec/skills/cycle/SKILL.md`

Git operations including `git merge --ff-only`, `git rebase`, `git branch -D`, and `git worktree remove --force` are described as prose-in-codeblock instructions that the LLM orchestrator must interpret and execute via its Bash tool. There is no script wrapping these: the LLM is expected to faithfully reproduce and execute multi-step git sequences. On misinterpretation, the result is branch corruption, orphaned worktrees, or lost commits. The stall-detection logic in `execute/SKILL.md:192-211` relies on the LLM correctly reading `git -C {worktree_path} log` and `git diff` output and making a branching decision.

### `git branch -D` used in non-recoverable position (low)

**File:** `/Users/cbobrowitz/Projects/loop-spec/skills/execute/SKILL.md:164,236`

Post-wave cleanup uses `git branch -D {task.worktree_branch}` (force delete) immediately after `git worktree remove`. If worktree removal fails mid-sequence, the branch is still deleted on the next attempt without checking whether the worktree was successfully detached. The orphan-prune block at line 235-236 also uses `--force` on `git worktree remove`. This is safe when all changes are merged, but the merged-ancestor check at line 234 could race with a concurrent operation in multi-session resume.

### `sync` used instead of `fsync` in state-write (low)

**File:** `/Users/cbobrowitz/Projects/loop-spec/lib/state-write.sh:43`

`sync` flushes all pending writes to disk at the OS level, which is correct but broader than needed. The comment says "fsync" but the implementation uses `sync`. On a system with a large write queue, this could cause an unnecessary performance stall. Not a correctness risk but a mismatch between intent and implementation.

### Stale state.json present in repo for an in-progress feature (low)

**File:** `/Users/cbobrowitz/Projects/loop-spec/.loop-spec/features/cycle-agent-teams/state.json`

A state file for `cycle-agent-teams` feature exists at `currentPhase: "discuss"` with `updatedAt: "2026-05-12T00:22:18Z"`. This is within the `stalenessHours: 48` window relative to creation date, but more than 48 hours old relative to today's date (2026-05-11 is the current date and this was created 2026-05-12 -- the timestamps indicate a clock issue or this is test data). The cycle skill's resume detection will present this to the user on next run as a resumable feature. This is a UX concern, not a correctness bug, but an abandoned in-progress state living in an untracked path creates noise for future runs.

## Fragile Areas

### Caller identity detection depends on transcript JSONL format (medium)

**File:** `/Users/cbobrowitz/Projects/loop-spec/hooks/restrict-agent-paths.sh:56-67`

The hook determines which subagent is writing by scanning the session transcript JSONL for the most recent `Agent` tool_use entry. This depends on the Claude Code harness producing stable JSONL with `type: "assistant"`, `message.content[].type: "tool_use"`, and `name: "Agent"` in the exact structure the Python parser expects. Any format change in a CC version update silently falls through to `caller = ""` (unrestricted). The broad `except Exception: pass` at line 63 means format changes produce no error or warning.

### design.md tier matrix is now stale (low)

**File:** `/Users/cbobrowitz/Projects/loop-spec/docs/design.md:157-172`

`docs/design.md` contains a tier matrix table (lines 157-172) that maps roles to tier-based model aliases (`opus`, `sonnet`, `haiku`). The 2026-05-11 preset-speed refactor moved model selection to `skills/shared/preset-matrix.md` and decoupled it from tier. `docs/design.md:182` still references `tier_matrix[role][state.tier]` as the dispatch lookup, which is now incorrect. The design doc is historical reference, but a contributor reading it would get a wrong mental model of how model selection works.

### map-codebase index.json fallback is fragile (low)

**File:** `/Users/cbobrowitz/Projects/loop-spec/skills/map-codebase/SKILL.md:40`

The `jq` pipeline that reads `.loop-spec/codebase/index.json` uses `|| echo '["arch"]'` as the entire fallback when `index.json` is missing or malformed. This means any parse failure silently maps all changed files to only the `arch` domain. Other domains (tech, quality, concerns, domain) are skipped until the next full re-map. The note at `skills/cycle/SKILL.md:240` says leaving index.json absent is intentional on first run, but the map-codebase skill does not distinguish "intentionally absent" from "corrupt".

### smoke.sh is the only automated end-to-end test (medium)

**File:** `/Users/cbobrowitz/Projects/loop-spec/tests/README.md:64`

The test matrix is defined as 36 cells (3 feature sizes x 3 tiers x 4 execution styles). Only cell #9 (trivial + quick + auto) is scripted in `tests/smoke.sh`. The remaining 35 cells are documented but not automated, requiring the claude CLI and authentication. Per `docs/design.md:615`, skill content is "not eval-tested at launch" and this is acknowledged as a known risk (Risk #15, Likelihood: High, Impact: High). The mitigation listed is dogfood + the one smoke cell. This is the primary test coverage gap.

### Retry budget exhaustion silently escalates without artifact cleanup (low)

**File:** `/Users/cbobrowitz/Projects/loop-spec/skills/execute/SKILL.md:124-128`

When `task.retries > executePerTask`, the skill pauses and escalates the task. Worktrees created for that task are not pruned until the orphan-prune step at the end of all waves. A stalled task with a dirty worktree will be skipped by the orphan pruner (line 228: "skipping prune"), leaving it on disk indefinitely without a recovery path documented for the user.
